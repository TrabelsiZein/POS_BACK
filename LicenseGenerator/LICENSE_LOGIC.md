# POS License System — Implementation Guide

This document describes the complete licensing logic used in the POS backend so it can be re-implemented in another application or language.

---

## Overview

The system uses **RSA-signed JSON license files** that are tied to a specific machine. The vendor generates a license file with a private key; the application verifies it using the matching public key at runtime. If the machine fingerprint in the license does not match the current machine, the license is rejected.

---

## Key Concepts

| Concept | Description |
|---|---|
| **installationId** | A SHA-256 fingerprint unique to the machine where the app is installed |
| **license.json** | A signed JSON file generated by the vendor and uploaded by the admin |
| **public_key.pem** | RSA public key embedded inside the application (verifies signatures) |
| **private_key.pem** | RSA private key kept secret by the vendor (signs licenses) |

---

## 1. Machine Fingerprint (installationId)

The `installationId` is computed **at application startup** and kept in memory.

### Algorithm (Windows)

```
Step 1 — Read MachineGuid from registry:
  reg query HKLM\SOFTWARE\Microsoft\Cryptography /v MachineGuid
  → e.g. "f81d4fae-7dec-11d0-a765-00a0c91e6bf6"

Step 2 — Read volume serial of system drive (e.g. C:):
  cmd /c vol C:
  → extract pattern like "1A2B-3C4D" (four hex digits, dash, four hex digits)
  Fallback: PowerShell WMI query if vol output cannot be parsed

Step 3 — Combine and hash:
  combined = normalize(machineGuid) + "|" + normalize(volumeSerial)
  normalize = trim + remove whitespace + toUpperCase

  installationId = SHA256(combined).toUpperCase()   [hex string, 64 chars]
```

**Example:**
```
MachineGuid   = "f81d4fae-7dec-11d0-a765-00a0c91e6bf6"
VolumeSerial  = "1A2B-3C4D"
combined      = "F81D4FAE-7DEC-11D0-A765-00A0C91E6BF6|1A2B-3C4D"
installationId = "E3B0C44298FC1C149AFB..." (64 hex chars)
```

## 2. License File Format

The license is a JSON file with two top-level keys: `data` and `signature`.

```json
{
  "data": {
    "company": "ABC Store",
    "installationId": "<64-char SHA-256 hex, uppercase>",
    "issuedAt": "2026-01-01",
    "expiresAt": "2027-01-01"
  },
  "signature": "<Base64-encoded RSA-SHA256 signature of the compact data JSON>"
}
```

### What is signed

The signature covers the **compact JSON string of the `data` object** (no extra spaces), encoded as UTF-8 bytes.

The data JSON used for signing must be in **fixed key order**:
```
{"company":"...","installationId":"...","issuedAt":"...","expiresAt":"..."}
```

## 3. License Generation (Vendor Side)

Done by the vendor using the `LicenseGenerator` CLI tool (or equivalent).

```
Inputs:
  - company name        (string)
  - installationId      (64-char hex obtained from the customer's machine)
  - expiresAt           (YYYY-MM-DD)
  - private_key.pem     (RSA PKCS#8 PEM, secret)

Steps:
  1. issuedAt = today's date (YYYY-MM-DD)
  2. Build compact data JSON with fixed key order
  3. Sign dataJson bytes (UTF-8) with RSA SHA256 using private key
  4. Base64-encode the signature
  5. Write license.json = { "data": {...}, "signature": "..." }
```

**Crypto algorithm:** `SHA256withRSA` (i.e., RSA + PKCS#1 v1.5 padding + SHA-256 digest)

---

## 4. License Upload & Validation (Application Side)

The admin uploads `license.json` via the UI. The backend validates it in this order:

```
Step 1 — Parse JSON
  Extract root.data and root.signature

Step 2 — Verify RSA signature
  Reserialize data as compact JSON string (UTF-8)
  Verify: signature(Base64-decoded) against dataJson bytes
  using public_key.pem (RSA SHA256)
  → Reject if invalid (tampered or not from trusted vendor)

Step 3 — Validate installationId
  licenseInstallationId = data.installationId
  machineInstallationId = MachineFingerprintService.getInstallationId()
  → Reject if they don't match (case-insensitive)

Step 4 — Parse and persist
  Extract company, issuedAt, expiresAt
  Mark any previous license as non-current
  Save new record to database with currentLicense = true
```

---

## 5. License Status Logic

Evaluated on every API request (or on demand):

```
Query current license (currentLicense = true):
  - None found          → MISSING
  - installationId mismatch with current machine → EXPIRED
  - daysUntilExpiry < 0 → EXPIRED
  - daysUntilExpiry <= 14 → WARNING
  - else                → VALID
```

**Warning threshold:** 14 days before expiry.

---

## 6. HTTP Enforcement (Request Filter)

A filter runs on every incoming API request:

```
If license status is MISSING or EXPIRED:
  → Return HTTP 402 with body:
    { "licenseError": true, "status": "MISSING" }   (or "EXPIRED")
  → Request is blocked

If status is VALID or WARNING:
  → Request passes through normally
```

### Bypass paths (always allowed, no license check)

```
/login
/config
/company-info
/license
/admin/license
/v3/api-docs
/swagger-ui
/thymeleaf
/franchise
/sse-endpoint
```

These allow the admin to log in and upload a license even when none exists.

## 7. Database Table: `license_record`

| Column | Type | Description |
|---|---|---|
| `id` | PK | Auto-generated |
| `company_name` | string | From license data |
| `installation_id` | string | From license data |
| `issued_at` | date | From license data |
| `expires_at` | date | From license data |
| `raw_json` | text | Full original JSON file |
| `uploaded_at` | datetime | When it was uploaded |
| `uploaded_by` | string | Admin username |
| `current_license` | boolean | Only one row is true at a time |

Rows are **never deleted** — they form a full upload history.

---

## 8. Public Key Storage

- Stored as `public_key.pem` inside the application's classpath at `/license/public_key.pem`
- Loaded once at startup
- Format: standard X.509 PEM (`-----BEGIN PUBLIC KEY-----`)

---

## 9. Summary Flow

```
[Vendor]
  1. Customer sends their installationId (from app startup log or admin UI)
  2. Vendor runs LicenseGenerator with installationId + company + expiry
  3. Vendor sends license.json to customer

[Customer / Admin]
  4. Admin uploads license.json via admin UI

[Application — on upload]
  5. Verify RSA signature against embedded public key
  6. Verify installationId matches current machine fingerprint
  7. Persist as current license

[Application — on every API request]
  8. Check current license status
  9. Block with HTTP 402 if MISSING or EXPIRED
```

---

## 10. Key Implementation Notes for Other Languages

- **RSA key format:** Private key is PKCS#8 PEM; public key is X.509 PEM
- **Signature algorithm:** `SHA256withRSA` = RSA PKCS#1 v1.5 + SHA-256
- **Data payload encoding:** UTF-8 bytes of the compact JSON string
- **Key order in data JSON is fixed** — must be exactly: `company`, `installationId`, `issuedAt`, `expiresAt`
- **installationId comparison** is case-insensitive
- **Machine fingerprint is Windows-specific** (registry MachineGuid + volume serial). Adapt for Linux/Mac if needed.
- The signature in the license file is **Base64 standard encoding** (not URL-safe)
